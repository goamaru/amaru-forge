# Forge Memory Management: MCP Proxy + Idle Session Reaper

**Status**: Approved design
**Date**: 2026-03-22
**Project**: Amaru Forge (`/Users/owner/Desktop/Tech Tools/amaru-forge`)

## Problem

Each Claude Code session spawned by Forge independently starts its own set of MCP servers (Playwright, Context7, Pinecone). With 8 concurrent sessions, this produces 32 duplicate MCP processes consuming ~3.1 GB — on top of the ~2.5 GB for the Claude processes themselves. Total Forge footprint: ~4.8 GB on a 48 GB machine.

The root cause is an impedance mismatch: Claude Code assumes it's the only instance (spawns everything fresh), but Forge's session model creates many concurrent instances. Additionally, idle tmux sessions hold their full memory footprint indefinitely because nothing reaps them.

## Solution

Two complementary features:

1. **Shared MCP Server Pool (Feature A)**: Forge spawns each MCP server once and proxies connections from all Claude sessions to the shared instances.
2. **Idle Session Reaper (Feature C)**: Forge detects idle sessions, captures hibernation snapshots, then kills the tmux session to reclaim memory.

Build order: Reaper first (simpler, immediate relief), then MCP proxy (structural fix).

## Feature A: Shared MCP Server Pool

### Architecture

Forge spawns one instance of each MCP server as a long-lived child process of the Forge main process. Each server communicates via a Unix domain socket. Claude Code sessions connect to these shared instances via wrapper scripts that bridge stdio to the socket.

```
Forge Main Process (Tauri)
├── mcp_pool module
│   ├── Context7 MCP server ↔ context7.sock   (shared, concurrent)
│   ├── Pinecone MCP server ↔ pinecone.sock   (shared, concurrent)
│   └── Playwright MCP server ↔ playwright.sock (shared, serialized)
│
├── tmux session 1 → Claude → wrapper scripts → sockets → shared servers
├── tmux session 2 → Claude → wrapper scripts → sockets → shared servers
└── tmux session N → Claude → wrapper scripts → sockets → shared servers
```

### Socket Layout

```
~/.terminal-forge/mcp/
├── context7.sock        # Unix domain socket
├── pinecone.sock        # Unix domain socket
├── playwright.sock      # Unix domain socket (mutex-guarded)
├── pids/
│   ├── context7.pid     # PID file for heartbeat checks
│   ├── pinecone.pid
│   └── playwright.pid
└── bin/
    └── npx              # Wrapper script that shadows real npx
```

### Wrapper Script Mechanism

Claude Code discovers MCP servers via plugins (`~/.claude/plugins/`) and explicit config (`~/.claude/mcp.json`). These configs specify commands like `npx -y @pinecone-database/mcp`. Forge cannot modify Claude Code's internal plugin resolution.

**Solution**: Forge generates a wrapper `npx` script in `~/.terminal-forge/mcp/bin/` and prepends that directory to PATH when creating tmux sessions. The wrapper:

1. Inspects the arguments to determine if this is a known MCP package launch.
2. If it matches a pooled MCP server (e.g., `@pinecone-database/mcp`, `@playwright/mcp`, `@upstash/context7-mcp`), connects to the corresponding Unix socket and bridges stdio ↔ socket using `forge-mcp-bridge` (a small Rust binary compiled as part of Forge's build).
3. If it does not match, passes through to the real `npx` binary.

```bash
#!/bin/bash
# ~/.terminal-forge/mcp/bin/npx (generated by Forge)
REAL_NPX="/opt/homebrew/bin/npx"
SOCKET_DIR="$HOME/.terminal-forge/mcp"
BRIDGE="$HOME/.terminal-forge/mcp/bin/forge-mcp-bridge"

# Check if this is a known MCP server launch
case "$*" in
  *"@pinecone-database/mcp"*)
    exec "$BRIDGE" "$SOCKET_DIR/pinecone.sock" ;;
  *"@playwright/mcp"*)
    # Playwright: acquire lock, connect, release on exit
    exec "$BRIDGE" --lock "$SOCKET_DIR/playwright.lock" "$SOCKET_DIR/playwright.sock" ;;
  *"context7-mcp"*|*"@upstash/context7-mcp"*)
    exec "$BRIDGE" "$SOCKET_DIR/context7.sock" ;;
  *)
    exec "$REAL_NPX" "$@" ;;
esac
```

### forge-mcp-bridge Binary

A small Rust binary (~50KB) compiled as part of Forge's build. It:

1. Connects to the specified Unix domain socket.
2. Bridges stdin/stdout ↔ socket bidirectionally.
3. With `--lock <path>`: acquires an exclusive `flock(2)` on the lock file before connecting, releases on exit. This provides serialized Playwright access without requiring the `flock` CLI (which is not available on macOS).

This eliminates external dependencies on both `socat` and the `flock` CLI — the bridge binary handles both stdio bridging and lock acquisition using Rust's `std::os::unix` and the `nix` crate (which is already a project dependency for the MCP pool module).

The bridge binary is compiled during `cargo build` as a separate binary target in `src-tauri/Cargo.toml` and copied to `~/.terminal-forge/mcp/bin/` at Forge startup.

### Statefulness Model

| Server | Statefulness | Sharing Strategy |
|--------|-------------|-----------------|
| Context7 | Stateless | Connection-pooled. Rust proxy accepts multiple clients, dispatches each to a server instance from the pool. |
| Pinecone | Stateless | Connection-pooled. Same as Context7. |
| Playwright | Stateful (browser context) | Serialized via `flock(2)` in `forge-mcp-bridge`. One session at a time. Lock released when the bridge process exits. |

### Server Lifecycle

**Startup**: MCP servers start lazily on first tmux session creation (not on Forge app launch). This avoids wasting memory if the user opens Forge but doesn't create sessions.

**Heartbeat**: A tokio background task checks every 30 seconds that each spawned server's PID is still alive. If a server has died, it is restarted and the socket re-bound.

**Shutdown**: When Forge quits, all MCP server child processes are killed and socket files are cleaned up. Implementation: add a handler in the existing `on_window_event` callback in `lib.rs` for `WindowEvent::CloseRequested` (or `Destroyed`). Alternatively, implement `Drop` on the `McpPool` struct held in `AppState` — Tauri drops managed state on exit.

**Crash recovery**: On Forge startup, check for stale socket files in `~/.terminal-forge/mcp/`. If socket files exist but the PID files reference dead processes, clean up the stale sockets before starting fresh.

### MCP Server Spawn Details — Connection-Pooling Proxy

**Key constraint**: MCP over stdio is inherently 1:1 — one client, one server process. A single server process cannot serve multiple clients simultaneously. Therefore, the Rust `mcp_pool` module acts as a **connection-accepting proxy** with an internal pool of server processes.

**Architecture per server type**:

```
                    forge-mcp-bridge (client 1)
                         │
                    Unix socket ──→ Rust mcp_pool::accept()
                         │              │
                         │         ┌────▼────────────────────┐
                         │         │  Pick or spawn server   │
                         │         │  from pool (max N)      │
                         │         └────┬────────────────────┘
                         │              │
                    forge-mcp-bridge    ▼
                    (client 2) ──→  Server instance stdin/stdout
                                    (1:1 relay per connection)
```

The Rust `mcp_pool` module for each server type:

1. `bind()`s a Unix domain socket at the designated path (e.g., `context7.sock`). Rust owns the listener — no MCP server process binds to the socket directly.
2. Maintains a pool of server process instances, each spawned with piped stdin/stdout.
3. On `accept()` of a new client connection:
   - Checks the pool for an idle server instance.
   - If none idle and pool size < max, spawns a new server process.
   - If pool is at max capacity, the connection blocks until a server becomes available.
   - Relays bidirectionally between the accepted client fd and the selected server's stdin/stdout.
4. When a client disconnects, the relay stops and the server instance is returned to the pool as "idle" (or killed if the pool is over its target size).
5. Idle server instances that haven't been used for 5 minutes are killed to reclaim memory.

**Pool sizing**:

| Server | Max pool size | Rationale |
|--------|--------------|-----------|
| Context7 | 3 | Stateless queries, moderate concurrency expected |
| Pinecone | 3 | Stateless queries, moderate concurrency expected |
| Playwright | 1 | Serialized access — `forge-mcp-bridge` acquires `flock(2)` before connecting, so only one client reaches the proxy at a time |

With 8 sessions but typical usage patterns (not all querying simultaneously), a pool of 3 keeps peak memory far below 8 dedicated instances while avoiding blocking in most cases.

### Environment Variables

The Pinecone MCP server requires `PINECONE_API_KEY`. Forge reads this from `~/.claude/mcp.json` (the `env` block in the server config). If not found there, falls back to the process environment (in case the user has it in `~/.zshrc`). The `mcp.json` file is the authoritative source since it's what Claude Code itself uses.

For plugins (Playwright, Context7), environment variables are inherited from the Forge process (which already runs `fix_env()` to set up PATH, TERM, etc.). No additional env vars are needed for these.

### PATH Injection

In `tmux.rs`, the `create_session` function is modified to prepend `~/.terminal-forge/mcp/bin/` to PATH:

```rust
// In tmux.rs::create_session(), before spawning:
let mcp_bin = home.join(".terminal-forge/mcp/bin");
let current_path = std::env::var("PATH").unwrap_or_default();
let new_path = format!("{}:{}", mcp_bin.display(), current_path);
// Pass to tmux via environment
Command::new(tmux_bin())
    .env("PATH", &new_path)
    .args(["-f", &conf, "new-session", "-d", "-s", name, "-c", cwd])
```

## Feature C: Idle Session Reaper

### Idle Detection (Hybrid)

Two signals combined:

1. **UI idle**: `last_accessed_at` timestamp on the Session struct. Updated when the user clicks into a session, types in the terminal, or edits session metadata. If `now - last_accessed_at > idle_timeout`, the session is a candidate for reaping.

2. **Activity guard**: Before reaping, two checks:
   - **Foreground command check**: Query `tmux display-message -t <name> -p '#{pane_current_command}'`. If the foreground process is a known active tool (`node`, `cargo`, `python`, `claude`, `git`, `npm`, `pnpm`), treat as active regardless of CPU. This catches processes that just started (recent CPU is low but work is happening).
   - **CPU fallback**: For unknown foreground commands, query CPU usage via `ps -o %cpu -p <pid>`. If CPU > 1%, skip. Note: `ps -o %cpu` reports lifetime-averaged CPU, which can miss short bursts — the command name check above covers that gap.

### Configuration

| Setting | Default | Location |
|---------|---------|----------|
| `idle_timeout_minutes` | 30 | Constant in `reaper.rs`. No UI yet. |
| Reaper scan interval | 60 seconds | Constant in `reaper.rs`. |
| CPU threshold | 1% | Constant in `reaper.rs`. |

Pinned sessions (`pinned: true`) are never reaped regardless of idle time.

### Hibernation Flow

When a session is selected for reaping:

1. **Capture snapshot**:
   - Working directory: `tmux display-message -t <name> -p '#{pane_current_path}'`
   - Git branch: existing `git.rs::get_branch(&directory)`
   - Foreground command: `tmux display-message -t <name> -p '#{pane_current_command}'`
   - Last 50 lines of scrollback: `tmux capture-pane -t <name> -p -S -50`
   - Timestamp of hibernation

2. **Store snapshot**: Write to the new `hibernate` field on the Session struct.

3. **Kill tmux session**: `tmux kill-session -t <name>`.

4. **Persist**: Save updated session store to disk.

5. **Notify frontend**: Emit `forge://session-hibernated` Tauri event with the session ID.

### HibernateSnapshot Struct

```rust
#[derive(Debug, Clone, Serialize, Deserialize)]
#[serde(rename_all = "camelCase")]
pub struct HibernateSnapshot {
    pub working_directory: String,
    pub git_branch: Option<String>,
    pub foreground_command: String,
    pub scrollback: String,        // last 50 lines
    pub hibernated_at: DateTime<Utc>,
}
```

Added to `Session`:
```rust
pub struct Session {
    // ... existing fields ...
    #[serde(default, skip_serializing_if = "Option::is_none")]
    pub hibernate: Option<HibernateSnapshot>,
}
```

**Note**: `#[serde(default)]` is required for backward compatibility — existing `sessions.json` files on disk won't have this field. `Option<T>` defaults to `None`.

**Frontend state**: The `SessionWithStatus` struct in `lib.rs` gains a `hibernated: bool` field (true when `hibernate.is_some()` and `!alive`). The sidebar uses three states: alive (green), hibernated (amber), dead (gray).

### Restore from Hibernation

The existing `restore_session` command in `lib.rs` already recreates a tmux session in the original directory. Changes:

1. If the session has a `hibernate` snapshot, include it in the response to the frontend.
2. Frontend shows the snapshot in the editor panel (working directory, branch, last command, scrollback) so the user has context.
3. Clear the `hibernate` field after restore.

### Session States

```
                    ┌──────────┐
         create ──> │  alive   │ <── restore
                    └────┬─────┘
                         │
                    idle + no CPU
                         │
                    ┌────▼──────────┐
                    │  hibernated   │  (tmux dead, snapshot saved)
                    └────┬──────────┘
                         │
                    user clicks restore
                         │
                    ┌────▼─────┐
                    │  alive   │
                    └──────────┘
```

The "dead" state (tmux died unexpectedly, no snapshot) remains as-is for sessions that crash outside the reaper flow.

### Reaper Implementation

A `tokio::spawn` background task started in `lib.rs::run()` → `.setup()`:

```rust
// Pseudocode
loop {
    sleep(60 seconds);
    let now = Utc::now();
    let store = state.store.lock();

    for session in &store.sessions {
        if session.pinned { continue; }
        if session.hibernate.is_some() { continue; } // already hibernated
        if !tmux::session_exists(&session.tmux_name) { continue; } // already dead

        let idle_minutes = (now - session.last_accessed_at).num_minutes();
        if idle_minutes < IDLE_TIMEOUT_MINUTES { continue; }

        // Activity guard
        if let Ok(cmd) = tmux::get_pane_command(&session.tmux_name) {
            if is_active_command(&cmd) { continue; } // known active tool
        }
        if let Ok(cpu) = get_pane_cpu(&session.tmux_name) {
            if cpu > 1.0 { continue; } // CPU active
        }

        // Hibernate and kill
        let snapshot = capture_snapshot(&session);
        session.hibernate = Some(snapshot);
        tmux::kill_session(&session.tmux_name);
        emit("forge://session-hibernated", &session.id);
    }

    save(&store);
}
```

### last_accessed_at Updates

Currently `last_accessed_at` only updates on explicit metadata changes. It must also update on:

- **Terminal input**: When `write_to_pty` is called (user typing in the session).
- **Session connect**: When `connect_session` is called (user clicks into the session).

These are the two Tauri commands that indicate active user engagement with a session.

**Debouncing**: `write_to_pty` fires on every keystroke. To avoid thrashing disk writes, `last_accessed_at` updates in memory only (skip the `save()` call). The reaper's periodic loop (every 60 seconds) flushes the in-memory timestamps to disk as part of its normal `save(&store)` call. This means `last_accessed_at` is at most 60 seconds stale on disk, which is fine for a 30-minute idle timeout.

## Out of Scope

- **Auto-detection of new MCP plugins**: If new Claude Code plugins are installed, they won't be automatically pooled. The wrapper script must be updated manually to include new MCP package patterns.
- **Per-session MCP config overrides**: All sessions share the same MCP servers. No per-project Pinecone indexes or per-session browser profiles.
- **Settings UI**: Idle timeout and pool configuration are Rust constants. No UI to change them.
- **Windows/Linux support**: Unix domain sockets are POSIX-only. Forge is macOS-only currently, so this is fine.
